# TGO 目标站对标（第一版）

## 1. 目的

这份文档用于回答一个核心问题：

我们现在做的，是不是已经在朝着 `https://tgo.infoq.cn/` 的产品形态靠近？

结论先行：

- 当前架构方向是对的
- 当前公开站与后台基础能力也是对的
- 但产品形态上，我们现在更像“内容 + 活动平台”
- 目标站更接近“科技领导者社区 / 学员组织平台”

也就是说，技术底座没有走偏，但信息架构、首页叙事、申请路径、会员体系表达，后续都需要继续向“社区型产品”靠拢，而不只是“媒体型内容站”。

## 2. 对标范围与依据

本次对标基于 2026-03-29 可公开访问的信息，主要来源：

- 目标站首页：`https://tgo.infoq.cn/`
- 目标站公开前端路由与页面文案（通过其公开 JS bundle 中的路由与文案字符串推断）
- InfoQ 官方 `TGO鲲鹏会` 频道页：`https://www.infoq.cn/TGO`
- 当前仓库文档与现有实现：`docs/route-map.md`、`docs/mvp-scope.md`、`apps/site/src/pages/*`、`apps/admin/src/views/*`

说明：

- 目标站部分页面存在登录态、会员态、动态接口与前端渲染逻辑，因此本版本对标以“公开可见结构”和“品牌表达”为主
- 部分运营数据口径在不同官方入口存在差异，例如首页出现“累计学员人数 2000+”，InfoQ 频道页出现“970+ 会员”，因此这类数字不能直接当作我们未来的固定文案，而应视为“动态运营指标位”

## 3. 最重要的判断

### 3.1 我们现在做的不是错，而是还不够像目标站

当前系统更突出：

- 主题
- 文章
- 活动
- 城市
- 申请
- 后台内容管理

这适合一个“内容运营平台”的 MVP。

但目标站真正强调的是：

- 科技领导者身份认同
- 学员价值与权益体系
- 分会 / 董事会 / 小组组织结构
- 严格筛选的加入流程
- 学员活动与会员关系联动
- 内容只是社区价值的一部分，而不是全部主体

因此，后续不能只继续堆文章和活动页，还要把“社区组织模型”补起来。

### 3.2 当前架构仍然适合继续推进，不需要推翻

现有架构：

- `apps/site`: Astro 公开站
- `apps/admin`: Vue + Vite 管理后台
- `apps/api`: Hono API
- `packages/db`: PostgreSQL + Drizzle

这个拆分依然适合目标站，因为目标站天然就是：

- 前台高品牌展示、高 SEO、强内容编排
- 后台高运营、高审核、高配置密度
- API 承载申请、活动、会员、组织、权限等业务规则

所以接下来应该做的是“产品结构校准”，不是“重选技术栈”。

## 4. 目标站公开信息架构概览

根据公开路由与导航信息，目标站至少包含这些一级能力：

### 4.1 首页

首页不是单纯的内容聚合页，而是品牌与招募页。

公开可见的核心表达包括：

- 品牌定位：`助力学员成为更优秀的科技领导者`
- 社区定位：`高端学习社区`
- 指标展示：累计学员人数、全年公开学习活动、覆盖技术团队规模、分布行业/领域
- 价值表达：首页存在明显的“加入我们”强 CTA
- 典型模块包括：
  - Hero 品牌区
  - 学员 / 大咖展示
  - 社区价值说明
  - 全球 / 城市覆盖展示
  - 活动展示
  - 合作 / 权益展示

### 4.2 学员展示

目标站导航中存在“学员展示”，至少包含：

- `/we/branches` 董事会成员
- `/we/members` 全球学员名单（会员可见或登录后可见）

这说明它不是只有“城市页”，而是有真实的人与组织结构展示。

### 4.3 学习活动

目标站活动体系比我们当前“活动列表 + 详情 + 报名”更完整，公开路由至少包含：

- `/event`
- `/event/:id`
- `/event/sign`

此外，从公开文案看，活动区分为：

- 私密小组活动
- 公开学习活动

并且活动与学员身份强相关，公开活动也强调审核、名额、学员占比。

### 4.4 申请加入

目标站把“申请加入”拆成两层：

- `/join`：招募与说明页
- `/apply`：正式申请表单页

这是非常关键的产品结构。

`/join` 负责：

- 讲清楚申请条件
- 讲清楚加入流程
- 讲清楚学员权益
- 回答常见问题
- 促成用户进入申请

`/apply` 负责：

- 真正收集线索与申请信息
- 进入审核流程

相比之下，我们当前只有一个 `/apply`，这在信息密度和转化设计上偏弱。

### 4.5 公告与关于

目标站公开路由至少还有：

- `/notice`
- `/notice/detail/:id`
- `/about/global`
- `/about/use`
- `/about/privacy`
- `/about/FAQ`
- `/about/finance`（部分能力可能与会员态关联）

说明它把“新闻公告”“全球介绍”“FAQ”“法律条款”“运营情况”都做成了独立内容域，而不是都塞进一个 about 页面。

### 4.6 学员中心与培训

从公开 bundle 可见还有：

- `/dashbord/info`
- `/dashbord/group`
- `/dashbord/activity`
- `/training/chart`
- `/training/teachers`

这说明目标站除了公开展示站，还有明显的“学员侧产品层”。

这部分不一定属于我们第一阶段 MVP，但必须纳入长期信息架构预留。

## 5. 我们当前与目标站的核心差距

## 5.1 一句话总结

我们已经有“发布平台骨架”，还没有“社区组织骨架”。

## 5.2 对标表

| 维度 | 目标站 | 我们当前 | 判断 |
| --- | --- | --- | --- |
| 品牌定位 | 科技领导者社区、学员组织、加入门槛明确 | 内容、活动、城市、申请平台 | 需要强化品牌与身份叙事 |
| 首页角色 | 品牌 + 招募 + 社区价值 + 活动 + 权益 | 内容集合首页 | 需要重做首页信息重心 |
| 申请路径 | `/join` 说明页 + `/apply` 表单页 | 仅 `/apply` | 明显缺失招募说明层 |
| 学员展示 | 有学员名单、董事会成员 | 无成员展示，仅内容与城市 | 缺少“人”的维度 |
| 组织结构 | 分会、董事会、小组、学员中心 | 城市页 + 后台角色权限 | 缺少业务域模型 |
| 活动体系 | 活动与学员身份、审核、签到联动 | 活动列表、详情、公开报名 | 基础有了，但社区联动不足 |
| 公告体系 | 独立公告列表与详情 | 暂无 `notice` 域 | 后续应补 |
| FAQ / 条款 | 独立 FAQ、隐私、使用条款 | 只有 about / privacy / terms | 基础有，但结构不完整 |
| 权益展示 | 明确展示加入后的收益清单 | 暂无权益体系页面 | 明显缺失 |
| 会员 / 学员中心 | 已有独立中心路由 | 暂无 | 非 MVP 必需，但需预留 |
| 培训体系 | 培训章程、培训师 | 暂无 | 可后置，但应纳入路线图 |
| 后台能力 | 目标站未知，但显然可驱动多类运营 | 我们已有内容、活动、资源、审核后台 | 我们的后台基础反而已走在前面 |

## 6. 对我们产品结构的直接影响

## 6.1 公开站不应只保留“内容型信息架构”

建议把公开站结构从当前：

- 首页
- 主题
- 文章
- 活动
- 城市
- 申请
- 关于

逐步调整为更接近：

- 首页
- 学员 / 董事会 / 分会
- 活动
- 公告
- 申请加入
- 关于 / FAQ / 条款
- 内容（可作为辅助，不一定始终占核心导航）

注意：

- 这不意味着要删除文章、主题、城市
- 而是意味着这些模块要重新从“社区服务”视角组织，而不是单独作为媒体栏目存在

## 6.2 当前的 `/topics` 与 `/articles` 不一定是目标站最优先的一级导航

我们目前把：

- 主题
- 文章
- 活动
- 城市

放在同一级导航里，这更像内容站。

目标站则更强调：

- 学员展示
- 学习活动
- 企业图谱
- 公告新闻
- 申请加入
- 关于 TGO

这意味着后续我们可能需要：

- 保留 `topics/articles` 作为内容域
- 但不一定继续让它们作为最显眼的一级导航
- 或者将其并入“内容 / 洞察 / 社区内容”下的二级结构

## 6.3 `/join` 应当成为下一批高优先级页面

如果目标是继续向目标站靠拢，最先应该补的公开页面不是更多文章，而是：

- `/join` 申请加入说明页
- `/faq` 或等价 FAQ 结构页
- `/notice` 公告列表页
- `/we/branches` 或等价的董事会 / 城市负责人展示页

其中 `/join` 的优先级最高，因为它直接影响：

- 首页 CTA 的落点
- 申请表单的转化率
- 用户对“为什么要加入”的理解
- 后续会员审核流程的合理性

## 7. 对数据模型与 API 的启发

目标站对我们最大的启发，不是视觉，而是业务域。

建议在现有模型基础上，预留或新增这些领域：

### 7.1 社区组织域

建议逐步加入：

- `chapters`：城市分会
- `chapter_board_members`：分会董事会成员
- `members`：正式学员
- `member_profiles`：学员公开资料
- `member_groups`：小组
- `member_group_bindings`：学员与小组关系
- `member_referrals`：推荐关系

### 7.2 加入流程域

建议把当前申请扩展为更完整的入会流程：

- `join_applications`
- `join_application_reviews`
- `join_interviews`
- `join_application_referees`
- `join_application_status_logs`

并支持更完整字段：

- 推荐人信息
- 技术工作年限
- 管理经验年限
- 团队规模
- 当前职位与公司阶段
- 目标城市 / 分会
- 加入诉求
- 审核意见 / 面试结论

### 7.3 活动域增强

在当前 `events` 基础上，可后续补充：

- `event_attendance_signins`
- `event_visibility_rules`
- `event_audience_types`
- `event_chapter_bindings`
- `event_member_quota_rules`

这样后续才支持：

- 公开活动 vs 学员活动
- 学员优先报名
- 签到码 / 活动签到
- 按分会运营活动

### 7.4 公开站配置域

建议让这些模块后台可配置：

- 首页指标数据
- 首页学员故事 / 推荐语
- 首页权益卡片
- 申请页 FAQ
- 分会负责人展示
- 公告位 / 顶部横幅

## 8. 对 MVP 的建议：不是全抄，而是先补关键差距

## 8.1 建议保留当前技术与工程分阶段策略

不要因为对标后发现差距较大，就把当前工程推翻重做。

更合理的方式是：

- 继续保留现有三应用架构
- 在当前 Phase 2 / Phase 3 结果上做产品结构迭代
- 按“更像目标站”的方向补域模型和页面

## 8.2 建议新增一个“社区化校准阶段”

在当前 MVP 之后，新增一段明确目标：

- 把当前站点从“内容运营 MVP”校准为“社区平台 MVP”

优先顺序建议如下：

1. 重构公开站首页文案与模块顺序
2. 新增 `/join` 申请加入说明页
3. 新增 `notice` 公告域
4. 新增 `branches` / `board members` 展示页
5. 扩展申请模型与后台审核字段
6. 再评估是否要引入学员中心和培训体系

## 8.3 哪些能力先不做

以下能力可以不进入下一轮 MVP，但应在架构与文档中预留：

- 完整学员中心
- 小组内部协作工具
- 会费支付与发票自动化
- 培训师与培训章程系统
- 企业图谱复杂图谱能力
- 会员专属内容权限

## 9. 设计层面的对标建议

对标目标站时，建议学的是“气质”，不是“像素”。

应该学习的方向：

- 高信任感
- 高门槛但不傲慢
- 强组织感
- 强人物感
- 强城市网络感
- 强申请转化引导

不建议直接照搬的方向：

- 具体视觉样式细节
- 所有历史包袱式页面
- 所有旧式前端交互形态
- 过多依赖图片堆砌的模块

我们更适合做成：

- 信息结构更清晰
- 后台驱动更强
- 视觉更现代
- 但品牌语气仍然偏“科技领导者社区”

## 10. 建议落地动作

基于这份对标，建议后续文档和实现按下面顺序推进：

### 10.1 先更新规划文档

优先补充：

- `docs/route-map.md`
  - 增加 `/join`
  - 增加 `/notice`、`/notice/[id]`
  - 评估 `/branches` / `/members` 路由
- `docs/data-model.md`
  - 增加 chapter / board member / member / join review 相关模型预留
- `docs/api-design.md`
  - 增加 join application、notice、chapter profile 相关 API 草案
- `docs/mvp-scope.md`
  - 明确“内容型 MVP”向“社区型 MVP”校准的边界

### 10.2 再推进公开站实现

建议优先级：

1. 首页改版
2. `/join` 页面
3. 公告页
4. 董事会 / 分会展示页
5. 申请表字段扩展

### 10.3 后台随后补齐

后台应补的不是“更多富文本”，而是：

- 分会管理
- 董事会成员管理
- 申请审核字段扩展
- 首页品牌模块配置
- FAQ / 公告配置

## 11. 结论

如果目标是“做一个类似 TGO 的平台”，那么：

- 我们的工程方向没有错
- 但产品重心需要从“内容发布”进一步转向“社区组织 + 申请转化 + 学员运营”

当前最关键的不是再补更多文章页，而是尽快补齐：

- 招募说明页
- 社区组织展示页
- 申请审核扩展模型
- 公告 / FAQ / 权益表达

只有这样，后续前台、后台和 API 才会真正向目标站的产品形态收敛。
